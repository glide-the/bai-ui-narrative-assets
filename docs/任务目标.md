## 1. 项目陈述（摘自 [`任务说明.md`](任务说明.md)）

- **项目名称**：洱海边的三重回声 —— 从白族文化到现代文明的视觉叙事。  
- **核心命题**：不是「把民族图案摆得更好看」，而是**让白族文化在现代界面中重新获得叙事位置**。  
- **体验形态**：单屏横向故事板；**主体页面禁止依赖长屏纵向滚动**；信息通过章节切换、符号弹层、详情面板与底部索引组织。  
- **符号三类**：身份（A）/ 生活（B）/ 仪式（C），每章仅呈现叙事所需子集，禁止「全符号图录式」堆砌。

---

## 2. 范围与非目标

### 2.1 必须交付（功能层面）

1. **七章叙事主流程**：每章可独立进入，具备统一的数据字段（标题、副标题、旁白、现代转折、关键词、场景符号列表等），与 [`完整叙事设计资产.md`](完整叙事设计资产.md) 主线一致。  
2. **五区布局能力**：Header（约 88px）+ 左侧叙事面板（约 360px）+ 场景区（约 980×660 @1440）+ 浮层详情 + 底部章节轨（约 92px），主视口基准 **1440×900**，见 [`UI设计资产说明.md`](UI设计资产说明.md)、[`design-system.md`](design-system.md)。  
3. **符号交互**：热点可聚焦/可点击；Tooltip 短提示；详情卡含编号、类别、叙事功能、转喻与适用场景。  
4. **关键章节交互**：第六章「传统民居 ↔ 现代楼房」**可拖拽对比**；第七章入口可进入**符号图谱**（按三类分组、可筛选、可返回叙事）。  
5. **「现代转折」信息层**：可按章节开关展示/隐藏，不破坏主叙事阅读节奏。  
6. **键盘与可访问性**：章节切换、图谱开关、对比滑块等具备键盘路径；焦点可见；状态**不完全依赖颜色**（对齐 `design-system` §5 与 UI 资产说明）。  
7. **状态持久化（浏览器侧）**：至少记住「上次章节」等键名应在单一模块集中定义（见 `design-system` §5.6），避免魔法字符串散落。  
8. **资产约束**：界面可见视觉须来自**本仓库生成或代码绘制**的资产包与 manifest；禁止外链图库、emoji 替代符号、占位摄影图作为交付态（与 `任务说明` §9、`工程设计稿` §4.2 一致）。

### 2.2 明确非目标（当前阶段）

- 不做「传统文化百科长文页」或廉价文旅促销页（见 `任务说明` §9）。  
- 不把「复制上游销售 CRM 页面」当作本叙事产品的目标（见 [`architecture/design_092_engineering_architecture.md`](architecture/design_092_engineering_architecture.md) §2.2、§4.4）。  
- 不在本文定义后端业务表结构；数据库与 API 以 [`architecture/项目架构设计说明.md`](architecture/项目架构设计说明.md) 为准。

## 4. 数据与配置契约（摘要）

以下内容在实现上须**数据驱动**（来自 `chapters.*` / `symbols.*` / `assetManifest` 或等价模块），字段命名可与 `工程设计稿` 示例对齐：

### 4.1 章节对象（每章至少）

- `id`（如 `chapter-01` … `chapter-07`）、展示序号、`title`、`subtitle`  
- `symbols[]`：本章出现的符号 ID 列表  
- `mainCopy`、`modernTransition`、`keywords[]`、可选 `mood[]`

### 4.2 符号对象（每个符号至少）

- `id`（A1…C8）、`category`（identity / daily / ritual）、中英文名称  
- `narrativeFunction`、`description`、`chapters[]`、`keywords[]`  
- `assetPath`（以 **manifest 为唯一权威**）

**ID 与资产命名**详见 [`design-system.md`](design-system.md) §3；与 CSV / mask 坐标文档交叉引用时保持同一 ID。

## 5. 验收标准（与 `任务说明` §10 对齐的「功能化」表述）

### 5.1 叙事与内容

- [ ] 能清楚读出「白族文化 × 现代文明」张力与七章推进逻辑。  
- [ ] 每章均有可映射到 UI 字段的旁白与现代转折，而非仅视觉装饰。  
- [ ] 符号出现符合章节叙事，而非全集陈列。

### 5.2 布局与交互

- [ ] 主叙事路径**无整页纵向滚动**（允许弹层/抽屉内局部滚动若已在 UI 规格中定义）。  
- [ ] 章节切换、符号详情、图谱、对比滑块、现代转折层均可完成闭环操作。  
- [ ] 键盘路径与焦点样式可感知；关键控件具备 `aria-label` 或与可见文案等价的信息。

### 5.3 视觉与设计系统

- [ ] 颜色与间距使用 `design-system` Tokens；材质感（米纸、墨黑、陶土红、石/布纹）与克制表达符合 UI 资产说明，避免廉价民俗化大渐变。

### 5.4 资产与工程诚实性

- [ ] `assetManifest`（或等价）可追溯到每个可见资产文件；禁止未声明的外链视觉依赖。

## 6. 与仓库工程架构的关系（[`architecture/项目架构设计说明.md`](architecture/项目架构设计说明.md)）

| 轨道 | 说明 |
| --- | --- |
| **叙事原型轨道** | 以 `index.html` + 静态资源可运行为目标时，遵循 `工程设计稿` 与本文 §2.1；不强制引入 Next.js 运行时。 |
| **Next.js 同仓轨道** | 仓库已存在 `app/(app)`、`app/api`、`app/lib` 等 PWA + Agent 能力；**叙事 UI 若迁入 App Router**，应复用路由与组件组织范式，但页面信息架构、文案与设计 Tokens 须回到本文与 `design-system`，不得照搬上游 CRM 业务语义（见 `design_092` §4.4）。 |
| **共享原则** | API 薄编排、协议与类型集中在 `app/lib`（尤其 `chat-schema`）——与叙事 UI 并行存在时，避免在 UI 内私自扩展私有协议字段。 |
